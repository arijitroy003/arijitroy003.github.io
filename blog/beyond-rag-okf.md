For the last three years, the default engineering response to any enterprise AI context problem was automatic: "Just build a RAG pipeline." Chunk your PDFs, generate embeddings, shove them into a vector database, and run semantic similarity searches at query time. It worked well enough for fuzzy, exploratory searches.

By mid-2026, the cracks in the RAG-everything approach are too wide to ignore.

### Where RAG Breaks

The failure modes are well-documented at this point:

- **Chunking destroys structure.** A 200-row pricing table split into 512-token chunks becomes incoherent fragments. The model reconstructs something plausible but wrong — the most dangerous kind of hallucination.
- **Retrieval is inherently probabilistic.** Cosine similarity finds text that *sounds* related, not text that *is* the answer. For compliance, legal, and financial use cases, "probably relevant" isn't good enough.
- **Synchronization is a nightmare.** When source documents update daily, re-embedding and re-indexing pipelines become their own engineering burden — often heavier than the original data pipeline.
- **Context windows grew faster than expected.** With 1M+ token windows now standard, the original motivation for RAG (fitting relevant context into a small window) is weaker.

RAG didn't fail. It just became the wrong default for a large class of problems.

### Enter Google's Open Knowledge Format

In June 2026, Google Cloud published OKF v0.1 — a vendor-neutral, open specification that formalizes what Andrej Karpathy called the "LLM Wiki" paradigm. The core insight:

> Instead of re-deriving knowledge at query time from raw chunks, store curated, cross-linked concepts that an agent reads and updates directly.

An OKF bundle is just a directory of markdown files with YAML frontmatter. No new runtime, no required SDK, no proprietary format. Each "concept" is a single `.md` file with one required field (`type`) and optional metadata (title, description, tags, timestamp). Files interlink with normal markdown links, creating a navigable knowledge graph.

The full v0.1 spec fits on a single page. That's the point.

### What Makes OKF Different

Three design principles separate OKF from yet-another-format:

**1. Minimally Opinionated.** The spec requires exactly one thing: a `type` field. Everything else — what types exist, what fields to include, how the body is structured — is left to the producer. The spec defines the interoperability surface, not the content model.

**2. Producer/Consumer Independence.** A bundle hand-authored by a human can be consumed by an AI agent. A bundle generated by a metadata pipeline can be browsed in a visualizer. A bundle synthesized by one LLM can be queried by another. The format is the contract; tooling at each end is independently swappable.

**3. Format, Not Platform.** OKF isn't tied to any cloud, database, model provider, or agent framework. It's just files on disk.

### RAG vs. OKF: When to Use Each

This isn't a binary replacement — it's a right-tool-for-the-job question:

**Use RAG when:**
- You're searching over a large, unstructured corpus where you don't know what questions will be asked
- The content is relatively static and the cost of probabilistic retrieval is acceptable
- You need fuzzy semantic matching ("find documents related to X")

**Use OKF when:**
- You need deterministic, accurate retrieval for structured knowledge (product catalogs, policies, compliance docs)
- The knowledge is actively maintained and evolves over time
- Multiple agents or tools need to share the same knowledge base
- You want human-readable, version-controlled knowledge that works in any editor

The pragmatic answer for most teams: use OKF as the curated knowledge layer for things you *know* matter, and RAG as the fallback for everything else.

### The Bigger Picture

OKF didn't appear in a vacuum. Karpathy's LLM Wiki gist (5,000+ stars in two months) proved the pattern. Obsidian users, CLAUDE.md authors, and anyone who's maintained a project's documentation in markdown already discovered the same thing: LLMs work better with curated, linked, maintainable markdown libraries than with repeated document search.

Google formalized the interoperability layer. The reference tooling they shipped — a BigQuery enrichment agent, a static HTML visualizer, and sample bundles — are deliberately minimal proofs of concept. The ecosystem of producers and consumers will grow far beyond what Google ships.

The shift from "embed everything and search" to "curate knowledge and navigate" is the same shift we saw from NoSQL-for-everything back to relational databases for structured data. The right abstraction wins, eventually.

OKF v0.1 is a starting point, not a finished standard. But the direction is clear: the future of AI context isn't better vector search — it's better knowledge.
